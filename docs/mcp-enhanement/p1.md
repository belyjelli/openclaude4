**MCP v2 Development Plan for openclaude4**  
**Version:** 1.0 (April 2026)  
**Status:** Full redesign – **no backward compatibility** with current `mcp.servers` in `openclaude.yaml` or the existing `internal/mcpclient` package.  
**Goal:** Turn the current basic stdio-only MCP client into a clean, production-grade, Claude-Code-level MCP system that is idiomatic Go, fits perfectly into the existing CLI + Bubble Tea TUI + core event system, and adds the missing “paste GitHub repo link → smart install with permission” feature you asked for.

### 1. Vision & Success Criteria
- **Full parity** with the advanced MCP system described in the markdown.engineering guide (multi-transport, rich config cascade, state-machine lifecycle, auth, live notifications, tool normalization).
- **Seamless smart installer** for any GitHub repo / npm package / raw command.
- **Zero friction for users** while keeping security-first (explicit permission at install + runtime).
- **Clean, maintainable, testable** Go code that respects the existing architecture (core/events, providers, TUI, Viper config layers).
- **Extensible** for future VS Code extension / gRPC bridge.

### 2. Design Principles (SOLID + Clean Architecture + Go Idioms)
| Principle              | How it is applied in MCP v2                                                                 |
|------------------------|---------------------------------------------------------------------------------------------|
| **Single Responsibility** | Each package does one thing: `config`, `transport`, `lifecycle`, `installer`, `proxy`      |
| **Open/Closed**        | New transports or auth flows added via interfaces without touching core manager            |
| **Liskov Substitution**| All transports implement the same `Transport` interface                                     |
| **Interface Segregation** | Small focused interfaces (`Connector`, `Authenticator`, `Notifier`)                       |
| **Dependency Inversion**| High-level `Manager` depends on abstractions, not concrete stdio code                     |
| **Clean Architecture** | Domain → Application → Infrastructure layers (no business logic in `cmd/` or `internal/tui/`) |
| **Go Idioms**          | `context.Context` everywhere, proper error wrapping, `testing` + `testify`, minimal dependencies |

### 3. New Package Structure (replaces `internal/mcpclient`)
```
internal/mcp/
├── config/          # new cascade loader + validation (v2 schema)
├── transport/       # interface + implementations (stdio, sse, ws, http, in-process…)
├── lifecycle/       # ServerConnection state machine + manager
├── auth/            # OAuth/PKCE + XAA support
├── installer/       # GitHub / npm / command smart installer
├── proxy/           # MCPTool adapter + name normalization
├── notifier/        # live ToolListChanged / ResourceListChanged handling
├── elicitation/     # structured input dialogs (TUI)

├── manager.go       # top-level MCPManager (orchestrator)

```

All new code lives under `internal/mcp/` – old `mcpclient` can be deleted after migration.

### 4. Config System v2 (breaking change – no backward compat)
New file: `mcp.v2.yaml` (or `~/.openclaude/mcp.yaml` with explicit version).

**Cascade (7 layers, exact order from guide, adapted to Go):**
1. Enterprise (locked `managed-mcp.json` – read-only)
2. Dynamic (CLI flag `--mcp-config`)
3. Claude.ai remote (future)
4. Project (`.mcp.v2.yaml` – CWD → root, child wins)
5. Local (per-project hash in `~/.openclaude/projects/`)
6. User (`~/.openclaude/mcp.yaml`)
7. Plugin-provided (namespaced)

**Features:**
- Environment variable expansion (`${HOME}` etc.)
- Allow/deny policies per server (name, command glob, URL)
- Deduplication by content signature (command + args or URL)
- Strict YAML schema with validation on load

Example snippet (new format):
```yaml
version: "2"
servers:
  - name: fs
    transport: stdio
    command: ["bunx", "-y", "@modelcontextprotocol/server-filesystem", "/Users/me"]
    approval: ask          # ask | always | never
    env: { ... }
    policies: { allow: ["read"], deny: ["write"] }
```

Old `openclaude.yaml` `mcp.servers` will be ignored with a clear deprecation warning on first run.

### 5. Core Components

**A. Transport Abstraction**
```go
type Transport interface {
    Connect(ctx context.Context, cfg *ServerConfig) (jsonrpc.Client, error)
    Close() error
}
```
Implementations: `StdioTransport`, `SSETransport`, `WSTransport`, `HTTPTransport`, `InProcessTransport`.

**B. Lifecycle Manager & State Machine**
`ServerConnection` with states: `Pending → Connecting → Connected → Failed → NeedsAuth → Disabled`.

- Batch connect (max 3 stdio + 20 remote)
- 30s timeout with race
- Exponential backoff reconnect (max 5 attempts)
- Live notification subscription
- Memoization by signature

**C. Tool & Resource Proxy**
- Automatic name normalization: `mcp__<server>__<tool>` (same rules as guide)
- Description capping, schema validation
- Rich result handling (images, binary → temp files)

**D. Auth Layer**
- Full OAuth2 PKCE + XAA support
- `needs-auth` state → inject pseudo-tool `mcp__<server>__authenticate`
- Token stored securely (keyring or OS keychain via `github.com/zalando/go-keyring`)

**E. Smart GitHub / Repo Installer** (your requested feature)
New command:  
`openclaude mcp install https://github.com/user/repo`  
or inside TUI: `/mcp install <url>`

**Flow (fully interactive with permission):**
1. Fetch repo (go-git or GitHub API)
2. Parse README + package.json for common patterns:
   - `bunx` / `npx` packages
   - Direct Go binary
   - Docker / raw command
3. Auto-suggest best command + name
4. Show diff of what will be added to config
5. **Explicit user permission** prompt (Y/n + approval policy choice)
6. Validate + add to correct config layer
7. Immediately attempt connect (dry-run mode optional)

This is the exact “paste link → done” experience you wanted.
